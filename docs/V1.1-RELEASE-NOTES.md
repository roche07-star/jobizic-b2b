## 🎉 주요 개선 사항

### 1. JD 수정/삭제 권한 체크 강화 🔒

**문제점**: PM이 다른 PM의 JD를 삭제할 수 있는 보안 이슈

**해결 방법**:
- UI 레벨: 본인/Owner/Admin만 삭제 버튼 표시
- API 레벨: DELETE/PATCH 엔드포인트 권한 검증

**영향 범위**:
- JD 목록 페이지
- JD 상세 모달

**코드 변경**:
- `app/jd/page.tsx`
- `app/api/jd/[id]/route.ts`

---

### 2. 후보자 소유권 개별 이전 기능 👥

**요구사항**: Owner가 특정 후보자를 다른 멤버에게 이전

**기능**:
- 후보자 상세에서 "소유권 이전" 버튼 (Owner만)
- 조직 멤버 선택 모달
- 후보자 + 파이프라인 함께 이전
- 성공 메시지 표시

**사용 시나리오**:
```
Owner: "이 후보자는 김PM님이 맡아주세요"
→ 소유권 이전 클릭
→ 김PM 선택
→ 이전 완료
```

**코드 변경**:
- `app/candidates/page.tsx`
- `app/api/candidates/[id]/transfer/route.ts` (신규)

---

### 3. 중복 후보 탐지 시스템 🚨

**문제점**: 같은 후보자가 여러 PM에게 중복 등록

**해결 방법**:
- DB 제약 조건: `UNIQUE (email, organization_id)`
- 프론트엔드 중복 체크
- 경고 다이얼로그 (등록자/등록일 표시)

**사용자 플로우**:
```
PM: 후보자 등록 (이메일: existing@test.com)
↓
시스템: "⚠️ 이미 등록된 후보자입니다
        이름: 홍길동
        등록자: 김PM
        등록일: 2026-06-05
        
        그래도 등록하시겠습니까?"
↓
PM: 취소 → 등록 안 됨 ✅
```

**코드 변경**:
- `migrations/add_unique_candidate_email.sql` (신규)
- `app/candidates/new/page.tsx`
- `app/api/candidates/check-duplicate/route.ts` (신규)

## ⚠️ 알려진 제한사항

### 1. 연락처 공개 범위
**현재**: 조직 내 모든 멤버가 모든 후보자 연락처 접근 가능  
**계획**: v1.2에서 연락처 승인 플로우 구현

**임시 대응책**: 
- 외부 Searcher에게는 민감 후보자 등록 자제
- Owner가 후보자 접근 로그 주기적 확인

### 2. JD 할당 시스템
**현재**: 모든 Searcher가 모든 JD 접근 가능  
**계획**: Searcher 10명+ 되면 JD push 할당 시스템 구현

**임시 대응책**:
- JD 제목에 담당자 명시 (예: "[김서처] Backend 개발자")
- 팀 회의로 담당 JD 구두 할당
